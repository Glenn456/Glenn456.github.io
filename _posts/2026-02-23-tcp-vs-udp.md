## Introduction

If you have ever streamed a video that buffered slightly but kept playing, and also sent an email that either arrived perfectly or not at all — you have already experienced the difference between **UDP** and **TCP** without knowing it.

These two protocols sit at the heart of almost every network communication you will ever analyse as a SOC analyst. Understanding how they work is not just academic — it directly shapes how you read packet captures, identify scanning activity, detect C2 beaconing, and investigate suspicious network behaviour.

In this post I will break down TCP and UDP from scratch, explain where and why each is used, and most importantly — show you what they look like in the wild from a security perspective.

---

## The Transport Layer — Where TCP and UDP Live

Both TCP and UDP operate at **Layer 4 — the Transport Layer** of the OSI model.

```
OSI Model
┌─────────────────────────┐
│  Layer 7 — Application  │  HTTP, DNS, FTP, SMTP
├─────────────────────────┤
│  Layer 6 — Presentation │  Encryption, encoding
├─────────────────────────┤
│  Layer 5 — Session      │  Session management
├─────────────────────────┤
│  Layer 4 — Transport    │  ← TCP and UDP live here
├─────────────────────────┤
│  Layer 3 — Network      │  IP, routing
├─────────────────────────┤
│  Layer 2 — Data Link    │  Ethernet, MAC addresses
├─────────────────────────┤
│  Layer 1 — Physical     │  Cables, signals, bits
└─────────────────────────┘
```

The Transport Layer is responsible for getting data from one application on one machine to the correct application on another machine. It does this using **port numbers**.

For example:
- Your browser on port 54321 talks to a web server on port 443 (HTTPS)
- Your email client on port 60001 talks to a mail server on port 25 (SMTP)

TCP and UDP both use ports — but they handle the actual delivery of data in completely different ways.

## TCP — Transmission Control Protocol

### The Core Idea: Reliability Over Speed

TCP is the **careful, reliable** protocol. It guarantees that every piece of data arrives at the destination, arrives in the correct order, and arrives without errors. If anything goes wrong in transit, TCP automatically retransmits the lost data.

Think of TCP like sending a package by tracked courier. You get a confirmation when it is picked up, updates along the way, and a signature when it is delivered. If it gets lost, the courier resends it.

### The TCP Three-Way Handshake

Before any data is exchanged over TCP, the two parties must establish a connection through a **three-way handshake**:

```
Client                          Server
  │                               │
  │ ──── SYN ────────────────────▶│   "I want to connect"
  │                               │
  │ ◀─── SYN-ACK ─────────────── │   "OK, I acknowledge, are you ready?"
  │                               │
  │ ──── ACK ────────────────────▶│   "Ready. Let's go."
  │                               │
  │ ════ Data flows both ways ════ │
  │                               │
```

- **SYN** (Synchronise) — client initiates, sends its sequence number
- **SYN-ACK** (Synchronise-Acknowledge) — server acknowledges and sends its own sequence number
- **ACK** (Acknowledge) — client confirms, connection is established

Only after this handshake completes does actual data transfer begin. This process adds overhead but guarantees both parties are ready and reachable.

### TCP Key Features

**Sequence Numbers**  
Every byte sent over TCP is numbered. If packets arrive out of order (which happens constantly on the internet), the receiver reorders them correctly using sequence numbers.

**Acknowledgements (ACK)**  
The receiver sends back acknowledgements confirming which data it has received. If the sender does not receive an ACK within a timeout period, it retransmits the data.

**Flow Control**  
TCP uses a **window size** to control how much data can be in transit at once. If the receiver is processing slowly, it shrinks the window to tell the sender to slow down. This prevents overwhelming the receiver.

**Congestion Control**  
TCP monitors network conditions and adjusts transmission speed to avoid contributing to network congestion — slowing down when the network is busy.

**Connection Termination — Four-Way Handshake**  
Closing a TCP connection requires a four-step process:

```
Client                          Server
  │ ──── FIN ────────────────────▶│   "I'm done sending"
  │ ◀─── ACK ─────────────────── │   "Got it"
  │ ◀─── FIN ─────────────────── │   "I'm done too"
  │ ──── ACK ────────────────────▶│   "Confirmed. Goodbye."
```

### Common Protocols That Use TCP

| Protocol | Port | Purpose |
|---|---|---|
| HTTP | 80 | Web traffic |
| HTTPS | 443 | Encrypted web traffic |
| SSH | 22 | Secure remote access |
| FTP | 21 | File transfer |
| SMTP | 25 | Email sending |
| RDP | 3389 | Remote Desktop Protocol |
| SMB | 445 | File sharing (Windows) |

## UDP — User Datagram Protocol

### The Core Idea: Speed Over Reliability

UDP is the **fast, lightweight** protocol. It sends data and does not look back. No handshake, no acknowledgements, no retransmission. If a packet gets lost — it is gone.

Think of UDP like dropping leaflets from a plane. You scatter them over an area and hope most land where intended. You have no idea which ones were picked up, and you are not going back to replace any that blew away.

### How UDP Works

UDP is remarkably simple. A UDP **datagram** (the UDP equivalent of a packet) contains:

```
┌──────────────────────────────────────────┐
│  Source Port     │  Destination Port     │
├──────────────────────────────────────────┤
│  Length          │  Checksum             │
├──────────────────────────────────────────┤
│                                          │
│              DATA (Payload)              │
│                                          │
└──────────────────────────────────────────┘
```

That is it. No sequence numbers. No acknowledgements. No connection state. The sender fires the datagram and moves on immediately.

### When is "No Reliability" Actually Better?

This sounds like a weakness — and for some use cases it is. But for others, reliability checking actually makes things worse:

**Real-time applications** — In a video call, if a packet carrying 20ms of audio arrives late, it is useless. It is better to just skip that 20ms and keep going than to wait for a retransmission. UDP lets applications do exactly this.

**High-frequency small queries** — DNS queries are tiny and happen constantly. Setting up a full TCP connection for each one would be massively inefficient. UDP's fire-and-forget approach is far faster for these small, rapid exchanges.

**Broadcasting** — UDP can send one packet to multiple recipients simultaneously (broadcast/multicast). TCP connections are strictly one-to-one.

### Common Protocols That Use UDP

| Protocol | Port | Purpose |
|---|---|---|
| DNS | 53 | Domain name resolution |
| DHCP | 67/68 | IP address assignment |
| SNMP | 161 | Network device monitoring |
| NTP | 123 | Time synchronisation |
| TFTP | 69 | Simple file transfer |
| VoIP/RTP | 5004/5005 | Voice and video calls |
| QUIC | 443 | Modern HTTP/3 transport |

> **Note:** DNS primarily uses UDP but switches to TCP when responses exceed 512 bytes (zone transfers, large records).

## TCP vs UDP — Side by Side

| Feature | TCP | UDP |
|---|---|---|
| Connection | Connection-oriented (handshake required) | Connectionless (no handshake) |
| Reliability | Guaranteed delivery | Best-effort, no guarantees |
| Ordering | Packets reordered if needed | No ordering |
| Error checking | Yes — retransmits lost data | Checksum only, no retransmission |
| Speed | Slower (overhead of reliability) | Faster (minimal overhead) |
| Header size | 20–60 bytes | 8 bytes |
| Flow control | Yes | No |
| Use cases | Web, email, file transfer, SSH | DNS, VoIP, streaming, gaming |

## Part 3: The Security Perspective — Why SOC Analysts Care

This is where it gets interesting. TCP and UDP behave very differently under attack, and knowing those differences lets you spot malicious activity faster.

### TCP SYN Scanning — The Attacker's Favourite Recon Tool

When attackers or penetration testers run Nmap to discover open ports, they typically use a **SYN scan** (also called a half-open scan):

```bash
nmap -sS 192.168.1.1
```

Here is what happens:

```
Attacker                        Target
  │ ──── SYN ───────────────────▶│
  │                               │
  │       Port OPEN:              │
  │ ◀─── SYN-ACK ─────────────── │  "Port is open, I'm listening"
  │ ──── RST ────────────────────▶│  Attacker resets — never completes handshake
  │                               │
  │       Port CLOSED:            │
  │ ◀─── RST ─────────────────── │  "Nothing here"
  │                               │
  │       Port FILTERED:          │
  │       [no response]           │  Firewall dropped the packet silently
```

**Why the RST?** By never completing the handshake, the scan avoids creating a full connection — making it quieter and faster. Many older systems did not log half-open connections.

**What to look for as a SOC analyst:**
- A single source IP sending SYN packets to dozens or hundreds of ports on a target in quick succession
- A pattern of SYN → SYN-ACK → RST across many ports — classic port scanning signature
- Nmap scans generate distinctive timing patterns that IDS systems like Snort and Suricata recognise

### TCP SYN Flood — Denial of Service

In a **SYN flood attack**, the attacker sends thousands of SYN packets but never completes the handshake:

```
Attacker sends:  SYN → SYN → SYN → SYN → SYN (thousands per second)
Server responds: SYN-ACK, SYN-ACK, SYN-ACK... (waits for ACK that never comes)
Result:          Server's connection table fills up — legitimate users can't connect
```

The server allocates resources for each half-open connection and holds them waiting for an ACK that never arrives. Eventually it runs out of connection capacity and crashes or becomes unresponsive.

**What to look for:**
- Massive spike in SYN packets from one or many sources
- Server's connection table approaching capacity
- High number of connections in `SYN_RECEIVED` state

### UDP Flood — Another DoS Vector

UDP floods work differently. Since UDP is connectionless, an attacker simply sends enormous volumes of UDP packets to random ports:

```
Attacker → UDP packet to port 12345 → Target
Target checks: is anything listening on port 12345? No.
Target sends: ICMP "Port Unreachable" back to attacker
Target checks: is anything listening on port 12346? No.
Target sends: ICMP "Port Unreachable"...
[Repeated millions of times per second]
```

The target exhausts its resources generating ICMP error responses. Bandwidth gets saturated and legitimate traffic can not get through.

### DNS Amplification Attack — Abusing UDP

One of the most common modern DDoS techniques exploits the fact that DNS uses UDP:

```
1. Attacker sends tiny DNS query (60 bytes) with spoofed source IP (victim's IP)
2. DNS server sends large DNS response (3000+ bytes) to the victim
3. Attacker sends thousands of these — victim receives massive amplified traffic
```

The attacker gets up to **50x amplification** — sending 1GB of queries generates 50GB of traffic at the victim. This is why open DNS resolvers are considered a security risk.

**What to look for:**
- Large volumes of DNS responses arriving at a host that did not send corresponding queries
- DNS responses significantly larger than typical (normal A record responses are tiny)

### UDP Port Scanning

UDP scanning is trickier than TCP because there is no SYN-ACK to confirm an open port:

```bash
nmap -sU 192.168.1.1
```

- **Open UDP port**: Often no response (application receives but does not respond)
- **Closed UDP port**: ICMP "Port Unreachable" response
- **Filtered**: No response (firewall dropped it)

Because UDP scanning relies on ICMP error messages (which are rate-limited by most OSes), it is much slower than TCP scanning. A full UDP scan can take hours.

### TCP for C2 Beaconing

Command and Control (C2) malware typically uses TCP because it needs reliable communication with the attacker's server. Common patterns:

- **Regular beaconing** — malware checks in with C2 every 60 seconds over TCP port 443 (HTTPS to blend in)
- **Long connections** — a persistent TCP connection kept open for hours or days
- **Unusual port usage** — C2 traffic on port 80 that does not look like HTTP, or port 443 traffic to an IP with no valid certificate

**What to look for:**
- Outbound TCP connections at perfectly regular intervals (beaconing)
- Persistent connections to external IPs that are newly registered or have poor reputation
- TCP connections on common ports (80, 443) where the traffic pattern does not match the expected protocol

## Reading TCP and UDP in Wireshark

Wireshark is the go-to tool for packet analysis. Here are the essential display filters every SOC analyst should know:

```bash
# Filter all TCP traffic
tcp

# Filter all UDP traffic
udp

# Filter by specific port
tcp.port == 443
udp.port == 53

# Show only SYN packets (port scanning detection)
tcp.flags.syn == 1 && tcp.flags.ack == 0

# Show TCP handshakes
tcp.flags.syn == 1

# Show RST packets (connection resets — anomaly indicator)
tcp.flags.reset == 1

# Show DNS queries only
dns && udp.port == 53

# Filter by source IP
ip.src == 192.168.1.105

# Combine filters
ip.src == 10.0.0.50 && tcp.flags.syn == 1
```

**Spotting a port scan in Wireshark:**  
Look for one source IP generating rapid SYN packets to sequential or random ports on a single target. The Info column will show `[SYN]` repeatedly, and you will see `[RST, ACK]` responses for closed ports.

## Final Thoughts

TCP and UDP are not just networking trivia — they are the foundation of everything you will investigate in a SOC. Every port scan, every C2 beacon, every DDoS attack, every suspicious connection in your SIEM is either TCP or UDP traffic at its core.

Understanding the handshake tells you when to expect SYN-ACK and when a RST is suspicious. Understanding UDP's connectionless nature explains why DNS amplification attacks work, and why UDP scanning behaves differently from TCP scanning.

The more fluent you become in how these protocols behave normally, the faster you will spot when something is wrong.
